# Gitリポジトリとは？

一言で言うと「**ソースコードの変更履歴をすべて記録しておく場所**」です📦

---

## 📖 わかりやすいたとえ話

**「セーブ機能付きのゲーム」**に例えると…

```
普通のフォルダ  → セーブできないゲーム
                  （上書きしたら前の状態に戻れない）

Gitリポジトリ  → セーブデータがいくつも残るゲーム
                  （いつでも前の状態に戻れる！）
```

---

## 🏠 リポジトリの種類は2つ

```
┌─────────────────────────────────────────┐
│  ローカルリポジトリ（自分のPC上）         │
│  c:\Users\toshi\product\timer-tukuru\   │
└────────────────┬────────────────────────┘
                 │ push（アップロード）
                 │ pull（ダウンロード）
                 ▼
┌─────────────────────────────────────────┐
│  リモートリポジトリ（インターネット上）   │
│  例: GitHub, GitLab など               │
│  ← クラウドのバックアップ場所           │
└─────────────────────────────────────────┘
```

---

## 🔄 基本的な作業の流れ

```
ファイルを編集する
      ↓
git add .          ← 変更を「保存前の準備」（ステージング）
      ↓
git commit         ← 変更を「記録（セーブ）」
      ↓
git push           ← GitHubに「アップロード」
```

---

## 📂 `.git` フォルダの正体

```
timer-tukuru/
├── .git/          ← ここにすべての履歴が入っている（触らなくてOK）
├── index.html
├── style.css
└── script.js
```

`.git` フォルダがあるフォルダ＝**Gitリポジトリ**です。

---

## 🚀 新しいプロジェクトを始めるとき

```bash
# ① フォルダを作って移動
mkdir my-project
cd my-project

# ② Gitを初期化（これが「Gitを指定する」作業）
git init

# ③ ファイル作成後、変更をステージング
git add .

# ④ コミット（記録）
git commit -m "最初のコミット"
```

---

## 💡 よくある状況

| 状況 | やること |
|------|---------|
| 全く新しいプロジェクトを始める | `git init` を実行 |
| GitHubからプロジェクトを持ってくる | `git clone URL` を実行（initは不要） |
| すでに `.git` フォルダがある | 何もしなくてOK！すでにGit管理されています |

---

## ✅ 用語まとめ

| 言葉 | 意味 |
|------|------|
| **リポジトリ** | 変更履歴を管理するフォルダ |
| **ローカル** | 自分のPC上のリポジトリ |
| **リモート** | GitHubなどのクラウド上のリポジトリ |
| **コミット** | 変更を記録する（セーブ） |
| **プッシュ** | ローカル → リモートへ送る |
| **プル** | リモート → ローカルへ取得 |
| **ステージング** | コミット前の「準備エリア」に変更を追加すること |
| **init** | 新しいリポジトリを初期化すること |
| **clone** | リモートリポジトリをローカルにコピーすること |
